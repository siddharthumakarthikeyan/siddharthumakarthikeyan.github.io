The Problem on the Shop Floor

Flexible Manufacturing Systems (FMS) rely on Autonomous Guided Vehicles to move materials between storage units, production lines, and warehouse areas. Get that logistics layer right and you get repeatable, low-error material flow at scale. Get it wrong and you get collisions, blocked aisles, and expensive downtime.

The challenge is that real manufacturing floors are not static. Humans move through them. Other vehicles stop and start unexpectedly. Production layouts change. The AGV needs to navigate all of this reliably — without an expensive sensor suite, without leaving its designated operational corridor, and without getting stuck.

Most existing approaches fall short in at least one of these dimensions. Line-following is cheap but completely inflexible. LiDAR-only odometry accumulates error over time. Camera-based SLAM is computationally expensive and sensitive to illumination. Classical planners like A* and Dijkstra plan in all available free space — they'll exit the virtual operating boundary the moment an obstacle forces a detour.

The real requirement isn't just obstacle avoidance — it's controlled obstacle avoidance: stay within boundaries, pass through required waypoints, and do it with a trailer attached.

The System: DDTMR — A Tow-Based AGV with a Trailer

We built and tested this on a Differential Drive Tow-Based Mobile Robot (DDTMR) — an AGV towing an off-axle trailer equipped with four caster wheels. This configuration is extremely common in real manufacturing (think of the tug-and-cart systems in automotive plants), but it introduces significant modelling complexity that most path planning papers ignore.

The trailer's four caster wheels swivel freely. As the AGV turns, each caster undergoes its own swivel dynamics — and as the coupling distance increases, the swivel effect amplifies. This caster-swivel coupling means a path that looks smooth for the AGV body can still cause the trailer to behave unpredictably. We derived the full dynamic model — kinetic energy, Lagrangian dynamics, Coriolis terms — explicitly accounting for each caster's swivel rate as a function of the AGV velocity and heading.

  • RPLidar A1M8 — 360° 2D LiDAR, 12 m range, sole navigation sensor

The Hybrid Navigation Stack

Rather than using a single planner, we designed a three-layer hybrid architecture. Each layer handles a distinct sub-problem at a different time horizon and spatial scale.

Layer 1 — Global Path Selection via Dynamic Programming

The shop floor is represented as a multistage graph of Key Points (KPs) — predefined via-nodes the AGV must pass through to move materials safely. The graph encodes costs based on distance between nodes, travel time, frequency of dynamic obstacles along a segment, and preferred routes.

We use Dynamic Programming with Bellman optimality to find the minimum-cost sequence of KPs from start to goal. The key advantage: because the AGV is constrained to navigate KP-to-KP, the accumulated localization error is corrected at each KP — preventing the drift that kills pure-odometry approaches over long runs. The DP cost map is also updated online when obstacles block segments, triggering a replan from the current KP.

Layer 2 — LiDAR-Based Obstacle Classification

Between KPs, the AGV uses only the LiDAR scan to classify obstacles in its immediate environment. The classification is based on time-history of laser scan returns:

  • Static obstacle — returns remain fixed over successive scans. The AGV switches to a modified TEB local planner and attempts a path within the virtual boundary. If no valid path exists, the AGV backtracks to the previous KP and replans.
  • Dynamic obstacle — returns move between scans (humans, other vehicles). The AGV computes a velocity adjustment factor Sa based on the obstacle's distance and approach direction, smoothly modulating speed to wait out or avoid the obstacle without leaving its operational corridor.

The velocity factor lookup covers 13 spatial regions per obstacle type (static, dynamic toward-center, dynamic away-from-center), calibrated empirically. This avoids the computational cost of re-running the full planner every time a human steps nearby.

Key design choice: Using LiDAR only — not camera, not fused sensors. The goal was a practical, low-cost solution deployable on budget AGVs. The RPLidar A1M8 costs a fraction of a depth camera and is illumination-independent, making it far more robust in real manufacturing settings where lighting varies throughout the day.

Layer 3 — Modified TEB Local Planner + MPC Controller

For executing the KP-to-KP trajectory, we use a modified Timed Elastic Band (TEB) planner with a custom cost function: 

Cost = w₁·Distance + w₂·Smoothness + w₃·ObstacleAvoidance
      + w₄·Velocity + w₅·Acceleration

The weights are tuned to prioritise boundary compliance over raw speed — ensuring the DDTMR stays within its virtual operating corridor even when avoiding obstacles. Trajectory execution is handled by a Model Predictive Controller (MPC) implemented in MATLAB, which takes the DP-generated KP sequence and computes the required motor torques, feeding back velocity commands to ROS.

AprilTag-based trailer localisation provides the trailer's pose estimate — the rear-facing camera detects tags mounted on the trailer front, giving real-time hitch angle and offset data used by the MPC to prevent trailer jackknifing during tight manoeuvres.

Simulation Experiments

We validated the approach in a 15 m × 15 m CoppeliaSim environment connected to ROS via a ROS–CoppeliaSim interface. The environment included:

  • Cube-shaped static obstacles representing machinery and production line segments
  • Human avatar models as dynamic obstacles, following paths that intersect the AGV's routes
  • 11 key points (A through K) as the navigation graph, with the DDTMR tasked to navigate A → I via optimal KP sequences
  • Virtual path boundaries defining the drivable corridor — violations count as failures

Path Comparison: Proposed vs A*, Dijkstra, TEB

The same scenario was run with four planners. The results are stark:

  • A* — plans in all available free space, frequently exits the virtual boundary, produces non-smooth paths that stress the trailer coupling
  • Dijkstra — similar to A* but additionally creates knots in the path when replanning around dynamic obstacles — a particularly dangerous condition for a tow-based vehicle where the trailer can jackknife
  • TEB alone — produces smooth paths but provides no guarantee of staying within the virtual operating boundary
  • Proposed DP–LiDAR–TEB — stays within the boundary in 97% of runs, produces the smoothest path, and replans cleanly when a static obstacle blocks the planned route entirely

Variance is the real metric for safety-critical systems. The proposed planner's variance in mission time (σ² = 3.61) is 4–12× lower than classical planners and 2–5× lower than TEB alone. In a manufacturing environment, unpredictable behaviour is often more dangerous than slow behaviour.

Robustness: Dynamic Obstacle Scenarios

We tested four specific dynamic obstacle scenarios to validate the velocity adjustment logic:

  • Case 1 — Obstacle approaches from the rear, enters the supercritical zone, then retreats. The planner smoothly ramps the velocity factor down and back up.
  • Case 2 — Obstacle enters gradually, reaches supercritical, then exits directly from it. The planner ensures no sharp velocity transitions — a smooth, gradual response.
  • Case 3 & 4 — Obstacle appears suddenly in a critical or supercritical region (a human who steps out from behind a machine). The AGV must react immediately without collision, while maintaining a smooth velocity profile to protect the trailer coupling.

In all four cases the planner generated appropriate velocity factors, the DDTMR avoided collision, and no boundary violations occurred.

Hardware Validation

Simulation results alone aren't enough. We built the actual DDTMR with trailer, created a physical test environment matching the CoppeliaSim layout, and ran all four planners on real hardware.

The hardware results mirrored the simulation findings:

  • A* and Dijkstra produced non-smooth paths with knots — particularly problematic when the trailer had to follow the same trajectory
  • TEB produced smoother motion but missed several KPs and exited the virtual boundary under dynamic obstacle pressure
  • The proposed planner successfully passed through all via points, produced smooth trajectories, and — when a static obstacle completely blocked the path between G and H — cleanly replanned to G→K→L→M→H without operator intervention

What This Means for Industry

The core contribution here is not a marginal improvement on a benchmark — it's a system architecture that specifically addresses the real constraints of industrial AGV deployment:

  • Single-sensor operation — LiDAR only, no cameras, no fusion complexity, lower cost, illumination-independent
  • Boundary compliance — the AGV stays inside its designated corridor even when evading obstacles, which is a hard safety requirement in real plants where production lines are on either side
  • Tow vehicle support — explicit modelling of caster-swivel dynamics means the trailer behaviour is predictable, not just the AGV body
  • Graceful degradation — when no valid path exists, the system backtracks and replans from the last safe KP rather than getting stuck or entering an unsafe region
  • Fleet-ready — validated with up to 10 simultaneous AGVs with zero boundary violations

Current limitation: At higher AGV speeds the trailer boundary constraint becomes harder to satisfy — the caster swivel dynamics have less time to settle. We're currently studying the speed threshold beyond which the virtual boundary guarantee no longer holds. More studies with different DDTMR speeds are needed before deploying at high-speed operation.
